# Reporte de analisis V1.0 - Calculadora ESDLA

Fecha de revision: 2026-04-29  
Estado analizado: version actual del workspace tras las correcciones aplicadas sobre el reporte V0.2.

## 1. Resumen ejecutivo

La version actual del proyecto esta en un estado sensiblemente mas estable que la revision V0.2. Los problemas estructurales mas graves detectados anteriormente han sido corregidos:

- Ya no hay IDs duplicados en `index.html`.
- Los grupos de radio de armadura estan separados por vista.
- La vista de Maniobra de Movimiento usa `vista-maniobra-movimiento` y no la errata anterior.
- Pifia MM y MM ya no comparten IDs ni grupo de radio.
- La logica de resistencia se ha centralizado en `assets/js/lib/resistencia.js`.
- Todos los JSON bajo `assets/tablas/` parsean correctamente.
- Existe una validacion estatica automatizada en `assets/otros/tests/validacion_estatica.ps1`.

El proyecto sigue siendo una aplicacion estatica sencilla, con una arquitectura suficiente para su tamano actual: `index.html` concentra la estructura de todas las vistas, `assets/js/app.js` inicializa modulos por vista, y las reglas se apoyan en tablas JSON externas.

Los riesgos principales restantes no son bloqueantes para una release, pero conviene tratarlos antes de que el proyecto crezca:

- No hay tests funcionales de calculo, solo validacion estatica.
- No se validan rangos de tiradas, niveles, distancias o valores negativos en muchos inputs.
- El README actual se llama `README.md` y su arbol interno ya referencia ese nombre.
- La aplicacion depende de CDN para Bootstrap.
- No se ha podido ejecutar `git` ni `node` en este entorno.
- La verificacion fue estatica; no hubo ejecucion real en navegador con recorrido de flujos.

## 4. Cambios positivos frente a V0.2

### 4.1 Separacion de radios de armadura

Antes todas las armaduras compartian `name="armadura"`. Ahora estan separadas:

- Ataque: `name="ataque_armadura"` en `index.html`.
- Hechizo dirigido: `name="hechizo_dirigido_armadura"` en `index.html`.
- Hechizo bola: `name="hechizo_bola_armadura"` en `index.html`.

El modulo de ataque tambien acota su busqueda:

- `assets/js/views/ataque.js`: usa `vistaAtaque?.querySelectorAll('input[name="ataque_armadura"]')`.

Impacto:

- Se elimina la interferencia entre Ataque y Hechizos.
- Limpiar Ataque ya no limpia armaduras de otras vistas.
- El calculo de ataque ya no puede leer un radio de armadura de Hechizo.

### 4.2 IDs duplicados corregidos en MM/Pifia

La vista de Maniobra de Movimiento usa ahora:

- IDs `mm_*`.
- Grupo `name="mm_dificultad"`.

La vista de Pifia conserva:

- IDs `pifia_mm_*`.
- Grupo `name="pifia_mm_seleccion"`.

El puente entre MM y Pifia queda en:

- `assets/js/views/mm.js`: `pifiasPorDificultadMm`.

Impacto:

- `document.getElementById(...)` vuelve a ser determinista.
- El popup de pifia puede preseleccionar correctamente la dificultad de pifia MM sin depender de IDs duplicados.

### 4.3 Resistencia centralizada

`assets/js/lib/resistencia.js` contiene:

- `calcularUmbralResistenciaTabla(...)`
- `ajustarNivel(...)`
- `calcularExcesoNivel(...)`

Lo usan:

- `assets/js/views/tr.js`
- `assets/js/views/hechizo.js`

Impacto:

- Se evita divergencia entre TR independiente y TR de hechizo basico.
- La logica de niveles superiores a 15 ya no reduce simultaneamente la fila del blanco por exceso del atacante.

### 4.4 JSON informativo renombrado

El archivo con errata fue corregido:

- Antes: `experiacia_nivel.json`
- Ahora: `experiencia_nivel.json`

La referencia actual esta en:

- `assets/js/views/informativas.js`

### 4.5 Validacion estatica incluida

El script `assets/otros/tests/validacion_estatica.ps1` cubre tres riesgos importantes:

- IDs duplicados.
- Patrones obsoletos.
- JSON invalidos.

Esto es un avance practico para evitar regresiones similares a las detectadas en V0.2.

Se corrigieron textos mojibakeados detectados en README, `index.html` y `hechizo.js`. El README se dejo en ASCII para evitar problemas de codificacion en GitHub y PowerShell.

### 5.3 No hay tests funcionales de reglas

Severidad: Media  
Archivos afectados: todos los modulos de calculo.

Existe validacion estatica, pero no tests que comprueben resultados concretos de reglas.

Ejemplos de casos no cubiertos automaticamente:

- Ataque con dado natural en tramo `sin_Modificador`.
- Ataque de garra/agarrar con maximo por tamano.
- Hechizo basico con dado natural 97-100.
- TR con niveles 0, 1, 15, 16 y 20.
- MM que produce `F` y preselecciona pifia.
- ME `leer runas` sin nivel de hechizo.
- Pifia con arma empunada montado.

Impacto:

- Cambios futuros en tablas o JS pueden romper calculos sin que el script actual lo detecte.
- La transcripcion de reglas depende de revision manual.

Recomendacion:

- Extraer funciones puras de calculo.
- Crear tests por dominio con casos de ejemplo del manual.
- Mantener el script estatico, pero complementarlo con tests de comportamiento.

### 5.4 Inputs numericos sin validacion de rango

Severidad: Media  
Archivos:

- `index.html`
- `assets/js/views/ataque.js`
- `assets/js/views/hechizo.js`
- `assets/js/views/critico.js`
- `assets/js/views/pifia.js`
- `assets/js/views/tr.js`
- `assets/js/views/mm.js`
- `assets/js/views/me.js`

Los inputs son `type="number"`, pero la logica acepta cualquier numero finito. No hay limites `min`, `max`, ni validacion de enteros.

Impacto:

- Se pueden introducir tiradas negativas, decimales o valores fuera de rango.
- Las tablas acaban devolviendo primera o ultima fila, lo cual puede ocultar errores de entrada.
- Un usuario puede obtener un resultado aparentemente valido con una entrada imposible.

Ejemplos:

- `critico_dados = -999` acabaria consultando la primera fila.
- `ataque_dados = 10000` acabaria consultando la ultima fila.
- Niveles o distancias decimales se aceptan parcialmente.

Recomendacion:

- Definir reglas de rango por campo.
- En HTML: usar `min`, `max`, `step`.
- En JS: validar enteros y rango antes de consultar tablas.

### 5.5 Dependencia de Bootstrap desde CDN

Severidad: Baja/Media  
Archivo: `index.html`

Bootstrap se carga desde:

- `https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/css/bootstrap.min.css`
- `https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/js/bootstrap.bundle.min.js`

Impacto:

- La app no es completamente offline.
- Si se usa en una partida sin conexion, la UI puede degradarse.
- El collapse del navbar depende del bundle JS externo.

Recomendacion:

- Incluir Bootstrap local en `assets/vendor/bootstrap/`.
- O documentar claramente que se requiere conexion para la version actual.

### 5.6 `inicio.js` depende de `bootstrap` global

Severidad: Baja/Media  
Archivo: `assets/js/views/inicio.js`

`colapsarNavbar()` llama:

```js
bootstrap.Collapse.getOrCreateInstance(...)
```

Si el script de Bootstrap no carga, `bootstrap` no existe y esa llamada puede romper al pulsar enlaces con navbar expandido.

Impacto:

- En entorno sin red o con CDN bloqueado, puede fallar la navegacion del navbar colapsado.

Recomendacion:

- Comprobar `window.bootstrap` antes de usarlo.
- O servir Bootstrap localmente.

### 5.7 Hechizos dirigidos: los maximos por subtipo limitan siempre la tirada modificada

Severidad: Media/Baja  
Archivo: `assets/js/views/hechizo.js`

La logica actual hace:

```js
const tiradaCapada = Math.min(tirada, maximosHechizoDirigido[subtipo] ?? 150);
const tiradaTabla = obtenerTiradaConsulta(tabla, dados, tiradaCapada);
```

Esto esta documentado con comentario y puede ser correcto, pero sigue siendo una regla sensible.

Impacto:

- Una tirada modificada muy alta queda limitada por subtipo.
- La tirada natural alta no tiene tratamiento equivalente al de hechizo basico/bola salvo que exista `sin_Modificador` en tabla.

Recomendacion:

- Confirmar contra el manual que el maximo por subtipo debe aplicarse antes de cualquier excepcion de tirada natural.
- Crear test de regresion con un caso por subtipo.

### 5.8 Hechizo basico y bola capan resultados modificados a 96

Severidad: Media/Baja  
Archivo: `assets/js/views/hechizo.js`

La funcion `calcularTiradaTablaHechizo(...)` aplica:

```js
if (dadosNaturales === 100) return 100;
if (dadosNaturales >= 97 && dadosNaturales <= 99) return dadosNaturales;
return Math.min(tiradaModificada, 96);
```

Impacto:

- Solo 97-100 naturales pueden consultar filas 97-100.
- Parece intencional y esta comentado, pero es una regla critica que deberia tener tests.

Recomendacion:

- Anadir casos de prueba: dado natural 96, 97, 99, 100 y dado 80 con modificador positivo alto.

### 5.9 Ataques naturales no abren pifia automatica

Severidad: Baja  
Archivo: `assets/js/views/ataque.js`

El codigo documenta:

```js
// Garra y agarre usan "Ataque fallido" en sus tablas, no "F"; no abren pifia automatica.
```

Impacto:

- Correcto si esa es la regla esperada.
- Si el manual contempla pifia de garra/agarrar, faltaria modelarla.

Recomendacion:

- Confirmar contra reglas.
- Si se confirma que no pifian, mantener como esta.

### 5.10 README.md simplificado

Severidad: Baja  
Archivo: `README.md`

El README ya usa `README.md` y un arbol ASCII simple.

Recomendacion:

- Mantener el arbol en ASCII simple si se edita de nuevo.

### 5.11 Falta validacion de referencias `new URL(...)` y rutas JSON desde tests

Severidad: Baja  
Archivo: `assets/otros/tests/validacion_estatica.ps1`

El script valida que los JSON sean parseables, pero no valida que todas las rutas usadas en `new URL(...)` existan.

Impacto:

- Un typo en una ruta JS podria pasar si el archivo JSON existe pero la ruta referenciada no se comprueba.

Recomendacion:

- Extender `validacion_estatica.ps1` para extraer rutas `../../tablas/...json` desde JS y comprobar existencia.

## 7. Seguridad y robustez

## 11. Conclusion

La version actual corrige los problemas estructurales importantes de la revision anterior y ya tiene una base mas fiable para release. La arquitectura es adecuada para una aplicacion estatica de este tamano, y el script de validacion estatica reduce el riesgo de regresiones DOM/JSON.

Ninguno parece impedir una V1.0, pero si el proyecto va a seguir creciendo, el siguiente paso tecnico deberia ser separar la logica pura de calculo de la manipulacion DOM y cubrirla con tests.
